# Release Notes: v1.9.3.6 (Major Update)

本次更新 (自 v1.9.1 升級) 涵蓋了底層資料庫架構的全面重構、核心交易演算法的深度校準，以及遠端遙控安全機制的全新設計。本次升級的最高指導原則為「回測與實戰邏輯的 100% 絕對鏡像」與「零延遲的終端控制」，徹底消除了過去版本中因非同步延遲、資料型態異常與防護機制分歧所導致的盤中實戰落差。

## 遠端通訊與安全驗證 (Remote Communication & Security)

* **去中心化動態金鑰綁定 (Dynamic OTP Pairing System)**
    重構 Telegram 授權機制。系統首次啟動時將於本地端動態生成 6 位數安全驗證碼 (PIN)，使用者需透過終端設備發送配對指令進行身分核實。驗證成功後系統會自動鎖定設備 Chat ID 並寫入本地設定檔，免除手動編輯配置檔的繁瑣，確保通訊渠道的絕對私密性。
* **零延遲本地輪詢架構 (Zero-Latency Local Polling)**
    廢除依賴第三方雲端 (如 Cloudflare) 的 Webhook 轉發機制。程式啟動時會主動發送指令解除雲端綁定，並全面接管通訊通道切換至本地長輪詢 (Long Polling) 模式。此架構不僅解決了高頻交易中最致命的網路傳輸延遲 (Latency)，更確保所有下單與參數查詢指令能以毫秒級速度直達本地交易核心。
* **執行緒防護鎖與狀態隔離 (Execution State Lock)**
    新增盤中狀態防護機制。在盤中監控引擎運作期間，系統將自動攔截並拒絕任何來自遠端的耗時運算請求 (如歷史回測、參數掃描或資料採集)，避免 CPU 資源搶占與 SQLite 讀寫衝突，確保實戰進出場邏輯的最高系統優先級。

## 核心架構升級 (Core Architecture Upgrades)

* **資料庫引擎重構 (SQLite Migration)**
    淘汰了早期的 JSON/CSV 檔案存取依賴，全面導入 SQLite 時序資料庫 (`quant_data.db`)。區分歷史回測 (`intraday_kline_history`) 與盤中實戰 (`intraday_kline_live`) 雙表架構，大幅降低 I/O 延遲，提升高頻讀寫時的穩定性與記憶體管理效率。
* **非同步併發數據採集 (Asynchronous Fetching)**
    盤中 K 線採集模組全面改寫，導入 `ThreadPoolExecutor` 進行多執行緒併發處理。結合自定義的 API 速率限制器 (Rate Limiter)，在不觸發券商端連線限制的前提下，將全市場數據更新延遲壓縮至極致。
* **記憶體快照攔截機制 (In-Memory Snapshot Interception)**
    針對實戰監控迴圈，新增記憶體鎖定與快照讀取機制。系統在進行條件邏輯判斷前會優先讀取最新的記憶體快照，徹底解決過去因等待資料庫非同步寫入而導致的「數據落後一分鐘」盲區。

## 交易邏輯與演算法對齊 (Algorithmic Synchronization)

* **回測與實戰引擎 100% 鏡像對齊**
    針對回測引擎與實戰引擎進行位元組級別 (Byte-level) 的邏輯同步，確保「盤中所見即回測所得」，消除任何演算法分支所造成的漏單或誤觸發現象。
* **嚴格洗盤計時重構 (Strict Wait-Time Enforcement)**
    修正了盤中實戰引擎提早觸發的瑕疵。將條件驗證嚴格對齊至使用者設定的等待時間閾值，並修正「突破前高」與「倒數計時」的先後驗證順序，確保領漲股在創新高時能精準中斷計時，防止過早放空強勢族群。
* **防誘空容錯演算法修復 (Pullback Tolerance Fix)**
    修復了盤中實戰引擎裡 DataFrame 序列評估失效的問題。全面實裝絕對高點容錯檢測，大幅增強系統在大震盪或假突破盤整期間的抗雜訊能力。
* **漲停板狀態機解鎖 (Limit-Up State Machine Optimization)**
    移除了盤中實戰專用的靜態漲停黑名單。改為採用與回測完全一致的相對狀態檢測 (動態比對前一分鐘高點與當前漲停價)，使系統具備捕捉「打開後二次漲停鎖死」的進場能力，同時完美兼容強勢股強制升級機制。

## 數據完整性與防禦性機制 (Data Integrity & Defensive Mechanisms)

* **強制型態安全防禦 (Strict Type Casting)**
    解決了 SQLite 偶發將數值欄位儲存為字串型態，導致浮點數比較靜默失效的嚴重例外。在進入核心邏輯矩陣前，強制對所有指標欄位執行嚴格的數值轉型，確保閾值觸發百分之百精確。
* **早盤動能基準修正 (Opening Volume Baseline)**
    優化了開盤初期 (09:00 至 09:02) 開盤均量尚未建立時的量能檢測邏輯。引入絕對成交量門檻作為備用檢測標準，確保第一分鐘的主力爆發訊號不會因除以零或缺失值而被略過。
* **動態漲停停損保護 (Dynamic Limit-Up Stop-Loss Protection)**
    在回測引擎中補齊了與實戰相同的「動態漲停保護」邏輯。系統在計算停損價差時，會自動計算對應股價的 Tick 階距，強制確保空單停損點不會設定在超過漲停價的位置，避免發生無法成交的死鎖狀態。
